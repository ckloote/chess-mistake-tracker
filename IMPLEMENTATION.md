# Implementation Plan

This document is the build plan for Claude Code. Each phase is scoped to be completable in one focused session and produces a working, testable increment.

## Ground rules for implementation

- **Read DESIGN.md first.** It is the source of truth for architecture and data model. If implementation reveals a flaw, update DESIGN.md in the same change.
- **Stay in the lane of the current phase.** Do not implement features from later phases even if they seem easy. Scope discipline keeps phases reviewable.
- **Tests for non-trivial logic.** Win% conversion, mistake detection, heuristic suggestions, and PGN parsing all get unit tests. UI tests are not required for MVP.
- **Commits per phase.** Each phase ends with a working build and passing tests.
- **No new dependencies without justification.** When you need to add one, note why in the commit message.
- **Prefer protocols over base classes.** `GameSource` and `Analyzer` are protocols, not ABCs.

## Phase 2: Lichess online ingestion

**Goal.** Pull the user's recent online games into the DB.

**Deliverables.**
- `backend/app/sources/base.py` — `GameSource` protocol and `GameRecord` dataclass.
- `backend/app/sources/lichess_online.py` — implementation. Hits `https://lichess.org/api/games/user/{username}` with `evals=true&clocks=true&pgnInJson=false&max=N&since=...`. Streams response, parses PGN per game using `python-chess`.
- `backend/app/services/ingestion.py` — orchestrates a source: fetches, dedupes against existing `(user_id, source, source_game_id)` rows, inserts new `Game` records. Sets `has_evals` based on whether PGN contains `[%eval ...]`.
- `POST /api/v1/games/import` endpoint accepting `{ source, since?, limit? }`.
- Tests for PGN parsing of a known sample game with evals and clocks.

**Acceptance.**
- `POST /games/import` with `{"source":"lichess_online","limit":10}` pulls 10 games for the configured user.
- Re-running is idempotent (same count of games in DB).
- A game with `%eval` annotations has `has_evals=true`.
- `GET /games?source=lichess_online` (stub endpoint OK for now) returns the imported games.

**Out of scope.** Studies, analysis, mistake detection.

## Phase 3: Lichess study ingestion

**Goal.** Pull configured Lichess studies and treat each chapter where the user is a player as a game.

**Deliverables.**
- `backend/app/sources/lichess_study.py`. Reads study IDs from config. For each: hits `https://lichess.org/api/study/{id}.pgn`, splits multi-game PGN, filters to chapters where user is white or black (matched by configured Lichess username, case-insensitive). Composes `source_game_id` as `{studyId}:{chapterId}`, where `chapterId` is the stable 8-char identifier from the chapter's `[Site "https://lichess.org/study/{studyId}/{chapterId}"]` header. (DESIGN.md uses chapterId; using a positional index would re-import duplicates whenever chapters are reordered.)
- Extend ingestion service to dispatch to the right source by name.
- `POST /api/v1/games/import` accepts `source="lichess_study"`.
- Test with a sample multi-chapter study PGN string (vendored in `tests/fixtures/`).

**Acceptance.**
- Importing a study brings in chapters where the user played.
- Chapters where the user didn't play are skipped (logged, not errored).
- Re-import is idempotent.

## Phase 4: Analysis pipeline — eval extraction

**Goal.** For games with embedded evals, populate `positions` with per-ply data.

**Deliverables.**
- `backend/app/analyzers/base.py` — `Analyzer` protocol, `EvalResult`, `PositionEval` dataclasses.
- `backend/app/analyzers/lichess_pgn.py` — parses `%eval` and `%clk` annotations from a PGN. Returns one `PositionEval` per ply with `eval_cp` (or `mate_in`) and `clock_ms`.
- `backend/app/services/analysis.py` — for a given `Game`: if `has_evals`, run `LichessPgnEvalAnalyzer`, populate `Position` rows (ply 0 through last move), compute `is_user_move` per ply, compute `time_spent_ms` from clock deltas. Set `analyzed_at`.
- `POST /api/v1/games/{id}/analyze` endpoint. Idempotent: re-running drops and recreates positions.
- `POST /api/v1/games/analyze-pending` runs analysis on all `has_evals=true` games not yet analyzed.
- Tests for eval/clock parsing across edge cases: mate evals, missing evals on some plies, decimal vs integer evals, clock formats.

**Acceptance.**
- A 60-move analyzed game produces 121 `Position` rows.
- `is_user_move` is correct for both colors.
- `time_spent_ms` is sane for at least 90% of moves (some early moves may have null clocks).

## Phase 5: Mistake detection

**Goal.** Identify mistakes per DESIGN.md detection rules.

**Deliverables.**
- `backend/app/chess_utils/winrate.py` — `cp_to_winrate`, `winrate_drop`, mate handling. Pure functions, well-tested.
- `backend/app/chess_utils/phase.py` — `is_endgame(board)` based on material count threshold; `is_quiet_position(board)` (no checks/captures available, no immediate threats); `detected_transition(board_before, board_after)` (queens off, pawn structure changed materially).
- `backend/app/services/mistake_detection.py` — iterates user moves in a game, computes win%-from-user's-perspective before and after, applies thresholds, applies suppression rules from DESIGN.md, sets time_pressure / endgame / transition flags. Creates `Mistake` rows.
- Detection runs at the end of the analysis pipeline (extend Phase 4's analyze endpoint).
- Tests covering: blunder/mistake/inaccuracy boundaries, color flipping (eval is white-relative; user is black), suppression rules, mate evals.

**Acceptance.**
- For a hand-curated test game with known mistakes, the detector flags exactly those mistakes at the expected severities.
- `GET /api/v1/mistakes?game_id=X` returns the detected mistakes (stub endpoint OK).
- No mistakes are flagged when `winrate_before < 30` and `winrate_after < 30`.

## Phase 6: Heuristic Layer A suggestion

**Goal.** Each `Mistake` gets a `suggested_step` and `suggestion_confidence`.

**Deliverables.**
- `backend/app/services/heuristics.py` — implements rules from DESIGN.md:
  - **Step 4** detector: from `P_after`, find opponent's best response. If forcing (check or capture in `python-chess`'s `Move` features) AND eval drops ≥200cp further, fire Step 4.
  - **Step 2** detector: from `P_before`, two paths (see DESIGN.md §"Step 2"). (a) `_step2`: engine's best move is a capture/check AND `M_user` non-forcing AND `winrate_before ≥ 50`. (b) `_step2_threat`: best move is *quiet* but creates a material/mate threat — detected by a null-move probe (play `M_best`, opponent passes, check whether the user then threatens mate or a SEE-winning capture). Path (b) needs the local engine and `chess_utils/see.py`.
  - **Step 1** detector: requires a "what if user passes" check. For MVP, approximate without a live engine: if engine's best move from `P_before` is responding to opponent's `M_opp` (e.g., it captures the piece `M_opp` just moved, or moves a piece `M_opp` just attacked), fire Step 1 — with a guard that it does *not* fire when `M_user` captured the same square (wrong-recapture ⇒ Step 3, not a missed threat). Note approximation in code comments — this is the heuristic most likely to mislead and is the strongest argument for adding local Stockfish in v1.1.
  - **Step 3** as default.
- Priority ordering 4 → 2 → 1 → 3, with all firing rules captured in a JSON `suggestion_debug` field on `Mistake`.
- Run heuristics at the end of mistake detection.
- Tests with hand-curated positions for each step.

**Acceptance.**
- Every `Mistake` row has a non-null `suggested_step`.
- The Step 4 case in the source article fires Step 4 (you'll need to manually re-create that position as a test case).
- The Step 2 case in the source article fires Step 2.

## Phase 13 (optional): Docker Compose path

**Goal.** Provide a documented containerized path for running the app on a different machine without installing `uv` or `nvm`.

This phase is optional and should be skipped unless the developer has a clear need (running on a home server, sharing with another person, etc.). It adds maintenance overhead and is not required for the MVP.

**Deliverables.**
- `Dockerfile` with multi-stage build:
  - Stage 1: `node:20-alpine`, copies `frontend/`, runs `npm ci` and `npm run build`.
  - Stage 2: `python:3.11-slim`, installs `uv`, runs `uv sync --no-dev`, copies backend code, copies `frontend/dist/` from stage 1.
  - Final command runs `uvicorn` serving both API and static frontend on port 8000.
- `.dockerignore` excluding `.venv`, `node_modules`, `data`, `.git`, etc.
- `docker-compose.yml` with one service, port 8000 published, `./data` mounted as a named volume so the SQLite DB persists across rebuilds.
- README section: "Running with Docker (optional)" with `docker compose up --build` and a note that this is not the daily dev workflow.

**Acceptance.**
- `docker compose up --build` produces a working app at `http://localhost:8000`.
- The DB persists across `docker compose down` and `docker compose up`.
- Image size is reasonable (under ~500MB compressed).
